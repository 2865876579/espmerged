# /code-check — 代码质量审查

## 核心原则

**发现 Bug → 列清单 → 等待确认 → 再动手改。**

永远不要跳过"列清单+等确认"这一步。哪怕只有一个 bug，也要先列出来让用户看到全貌。

## 审查范围

当用户调用 `/code-check` 时，你需要审查当前项目中的**用户代码**（跳过 `build/`、`managed_components/`、`__pycache__/`、第三方库）。

### ESP32 固件项目 (espcodex)
- C 文件：`main/*.c`, `main/*.h`, `components/*.c`, `components/*.h`
- 构建配置：`CMakeLists.txt`, `sdkconfig.defaults`, `partitions.csv`
- 关注：内存泄漏、栈溢出、空指针、线程安全、FreeRTOS 正确用法、ESP-IDF 最佳实践

### Python 服务端项目 (agentcodex, espagent)
- Python 文件：`server/*.py`
- 关注：异常处理遗漏、资源未释放、并发安全、冗余逻辑、类型安全

## 审查流程（必须严格遵守）

### 第一步：扫描与分析

1. 读取所有用户代码文件
2. 对比参考项目——小智 AI 聊天机器人 [`78/xiaozhi-esp32`](https://github.com/78/xiaozhi-esp32)：
   - **ESP32 C 代码**：重点参考小智项目的代码组织方式、ESP-IDF API 用法、FreeRTOS 任务设计、内存管理策略、错误处理模式
   - **音频处理**：参考小智的 Opus 编解码、音频流水线设计
   - **WebSocket 通信**：参考小智的连接管理、重连机制、心跳保活
   - **Python 服务端**：参考小智服务端的 LLM 调用、TTS/STT 集成模式
3. 检查以下维度：
   - 🐛 **Bug**：逻辑错误、空指针/None 解引用、缓冲区溢出、竞态条件、资源泄漏
   - 🔁 **冗余**：重复代码、无用变量、死代码、过度嵌套、重复逻辑
   - 📐 **不规范**：命名不一致、缺少错误处理、硬编码魔法数字、注释缺失、函数过长
   - ⚡ **性能**：不必要的内存分配、阻塞调用、低效算法
   - 🔒 **安全**：输入未校验、缓冲区无边界检查、SQL 注入（如有）、路径遍历

### 第二步：输出 Bug 清单（⚠️ 只列不改）

用以下格式输出，按严重程度排序：

```
## 🔍 代码审查报告

### 🔴 严重（会导致崩溃/数据丢失）
| # | 文件:行号 | 问题描述 | 影响 |
|---|----------|---------|------|
| 1 | main.c:42 | xxx | xxx |

### 🟡 一般（功能异常/潜在风险）
| # | 文件:行号 | 问题描述 | 建议修复方式 |
|---|----------|---------|-------------|

### 🟢 建议（代码质量/可维护性）
| # | 文件:行号 | 问题描述 | 建议修复方式 |
|---|----------|---------|-------------|

### 🔁 冗余代码（可简化/合并/删除）
| # | 文件:行号 | 问题描述 | 简化建议 |
|---|----------|---------|---------|

### 📐 不规范代码（命名/风格/结构）
| # | 文件:行号 | 问题描述 | 规范建议 |
|---|----------|---------|---------|
```

### 第三步：等待用户确认

输出清单后，**必须**询问用户：

> 以上共发现 **N** 个问题。请告诉我要修复哪些：
> - "全部修复" — 修复所有问题
> - "只修复严重" — 只修复 🔴 严重问题
> - "修复 1,3,5" — 按编号修复指定问题
> - "先修严重的，建议的再看" — 分步来
> - "不用改，只是看看" — 退出

**在用户明确回复之前，不修改任何代码。**

### 第四步：得到确认后才修改

- 一次只改一类问题，改完让用户确认
- 每改完一个文件，说明改了什么
- 如果修复引入了新问题，重新列出

## 小智项目参考要点

审查 ESP32 固件代码时，重点对照 [78/xiaozhi-esp32](https://github.com/78/xiaozhi-esp32)：

| 方面 | 参考要点 |
|------|---------|
| **项目结构** | main/ 放应用代码，components/ 放可复用组件，每个组件有独立 CMakeLists.txt |
| **内存管理** | ESP32 内存有限，避免 `malloc` 碎片化，优先静态分配或 `calloc` 一次性分配 |
| **FreeRTOS 任务** | 任务栈大小合理（音频任务 ≥4096），使用 `xQueue`/`xEventGroup` 做任务间通信 |
| **音频流水线** | 采集→编码→发送 流水线设计，用 RingBuffer 或 Queue 传递数据，避免拷贝 |
| **WebSocket** | 断线自动重连，心跳保活，发送/接收分离到不同任务 |
| **错误处理** | ESP_ERROR_CHECK 宏统一处理，关键路径有日志，错误码逐层向上传递 |
| **WiFi 管理** | 连接状态机完整（断开→扫描→连接→获取IP→就绪），有超时和重试 |
| **代码风格** | Google C++ Style（小智项目采用），命名清晰，函数单一职责 |

## 注意事项

- ⚠️ **绝对不要**跳过清单直接改代码。这是本 skill 最重要的规则。
- 如果项目代码很少（比如只有几个文件），也要完整走一遍流程。
- 审查第三方库代码（`managed_components/`、`78__*` 等）只在你被明确要求时才做。
- 修复代码后，如果有 `build_flash.bat` 等编译脚本，提醒用户可以编译验证。
